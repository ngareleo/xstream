# Key Invariants — Never Violate These

1. **All SQL goes through `db/queries/`** — no `getDb().prepare(...)` calls outside that directory.
2. **GraphQL schema changes require `bun relay:compile` in `client/`.** `__generated__/` artifacts are gitignored; a stale artifact breaks Relay queries at runtime.
3. **`SourceBuffer.appendBuffer()` must never be called while `updating === true`.** Always `await waitForUpdateEnd()` first — violation throws `InvalidStateError` and breaks MSE.
4. **Init segment is the first frame on every new stream connection.** Server sends `init.mp4` before any `.m4s`; client appends it before any media segment. Order broken = decoder can't initialise.
5. **`path` is the unique key for libraries and videos.** Two libraries can share `name`; only `path` is unique.
6. **`MediaSource.endOfStream()` must be called when streaming finishes.** `BufferManager.markStreamDone()` handles it — skipping it stalls `<video>`.
7. **Revoke object URLs on teardown.** `BufferManager.teardown()` calls `URL.revokeObjectURL()`. Always teardown on unmount or resolution switch.
8. **`content_fingerprint` is `NOT NULL`.** Old `tmp/xstream.db` without this column must be deleted and regenerated — no backward-compatible migration.
9. **Relay global IDs must be URL-encoded in route links.** Global IDs are base64 and may contain `/`, `+`, `=`. Use `encodeURIComponent(id)` on the way in, `decodeURIComponent` (or `resolveVideoId`) on the way out.
10. **One resolver owns each GraphQL field.** async-graphql derives types and resolvers from struct attributes — multiple resolver definitions for the same field are a compile-time error.
11. **Playback-path resolvers return a typed union for known failure modes — they never throw a plain `Error`.** `start_transcode` returns `StartTranscodeResult = TranscodeJob | PlaybackError`. The resolver wraps only genuinely unexpected failures as `INTERNAL`; cap rejection, video-not-found, probe failure, and encode failure are all discriminated `PlaybackError` variants. Mid-job failures must populate `ActiveJob.error_code` **before** notifying subscribers — the subscription delivers the final state to the client in the same push, so setting it after is a race.
12. **The network stream is a sink the consumer pulls from — never a source the producer pushes into.** `server-rust/src/routes/stream.rs` uses per-request segment fetching via `axum::Body::from_stream` over an `mpsc::Receiver`. Each client read request delivers exactly one segment (init on the first call; media segments by index after). There is no internal loop, no hidden queue between disk read and network send. An encoder-wait inside the request handler is fine; spawning a background task that eagerly streams is not.
13. **Transition state belongs to the caller doing the transition, not the dispatcher.** One-shot boolean guards (`prefetchFired`, `firstFrameRecorded`, `firstPrefetchRecorded`) are reset by the lifecycle caller — `handleChunkEnded`, `handleSeeking`, `handleMseDetached`, `switchResolution`, or the initial `startPlayback` class construction — never inside `startChunkSeries` or other dispatcher functions that are reached via async continuations (e.g. `buf.seek().then(startChunkSeries)`). Resetting in both the caller and the dispatcher creates a double-reset race: the RAF can fire in the async gap after the caller's reset but before `startChunkSeries` runs, and a second RAF fires after the dispatcher resets again. Apply this pattern to all single-fire per-session flags. See trace `b3dbbc34…` for the seek-time duplicate-prefetch that exposed this.

14. **No `expect`, `unwrap`, or silently-discarded errors in production Rust code.** The whole reason this project is moving to Rust is to use the type system for edge-case coverage. Every `expect` opts out of that. Concretely, in `server-rust/src/**` (excluding `#[cfg(test)]` blocks):
    - **No `.unwrap()`, no `.expect("…")`.** Even on a literal that "can't" fail to parse, propagate via `?` so the type system makes the failure path visible. Test code may use `.expect("clear message")` — every panic still has a documented reason.
    - **No `let _ = fallible_call();`.** Either propagate (`?`), handle inline (`if let Err(err) = … { tracing::warn!(…) }`), or — when the Result genuinely cannot fail — choose an infallible alternative (e.g. `format!`/`push_str` instead of `write!` to a `String`).
    - **Mutex poisoning is a typed error, not a panic.** `Db::with` returns `DbResult<T>`; poisoning surfaces as `DbError::PoisonedMutex` so resolvers `?`-propagate it into `async_graphql::Error` rather than crashing the worker thread.
    - **Silent JSON / parse fallbacks are forbidden when the data was supposed to be well-formed.** Bad JSON in a DB-written column is corruption — `tracing::warn!` with `library_id`/`video_id` and the raw value, then degrade visibly (e.g. empty list). Distinct from an `Option::unwrap_or_else` fallback where `None` is a valid documented state (e.g. `video.title.unwrap_or_else(|| video.filename)`); those stay.
    - **Top-level `main()` returns `AppResult<()>`.** Init failures (DB open, OTLP exporter build, port bind, signal-handler install) propagate out of the process via `Err`, with the `#[source]` chain printed to stderr so a "caused by" trail is visible without `RUST_BACKTRACE`.
    - **Two layers of error type:** module-level (e.g. `DbError`) for everything that can fail inside one subsystem, plus a top-level `AppError` for things that bubble out of `main`. `thiserror` for both; `From` conversions where one wraps another.
    - **Every error is also logged via `tracing::error!` so it lands in Seq with the request TraceId.** Propagation alone isn't enough — a typed error returned to the GraphQL client is invisible to the operator unless it's also emitted through the OTel pipeline. For resolver errors this happens via the `ErrorLogger` async-graphql extension (`server-rust/src/graphql/error_logger.rs`), which fires `tracing::error!` from inside the per-request `http.request` span; `tracing-opentelemetry` then attaches the inbound W3C trace context, so the Seq event correlates 1:1 with the originating request. For startup failures, `main()` emits both `tracing::error!` (OTel → Seq, no TraceId since there's no request) and a stderr `caused by:` chain (for systemd / CI / no-OTel environments). New error sites must follow this two-channel pattern; an error returned but not logged still counts as swallowing for the operator.
